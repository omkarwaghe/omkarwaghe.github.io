Step-by-Step Guide for developing BI report with App Engine

  • Create Data Source in Application Engine:
    • Develop an Application Engine program to extract data from the PeopleSoft database.
    • Use SQL or PeopleCode to fetch the required data and store it in a temporary table or a state record.
  • Register Data Source in BI Publisher:
      Navigate to Reporting Tools > BI Publisher > Data Source. Register the data source created by your Application Engine program.
  • Design BI Publisher Report Template:
    • Use tools like Microsoft Word with the BI Publisher plugin to design your report template.
    • Define the layout and format of the report, and map the data fields from your data source to the template.
  • Upload and Register the Template:
    • Go to Reporting Tools > BI Publisher > Report Definition.
    • Create a new report definition and upload your template.
    • Link the data source to the report definition.
  • Configure Application Engine to Call BI Publisher:
  • In your Application Engine program, add a PeopleCode step to call the BI Publisher report.
  • Use the XML Publisher API to generate the report. Example PeopleCode snippet:


    import PSXP_RPTDEFNMANAGER:*;
    Local PSXP_RPTDEFNMANAGER:ReportDefn &ReportDefn;
    &ReportDefn = create PSXP_RPTDEFNMANAGER:ReportDefn("REPORT_DEFN_ID");
    &ReportDefn.SetRuntimeDataXML(&XMLDoc);
    &ReportDefn.ProcessReport("", "");

  • Schedule and Run the Application Engine:
    • Schedule the Application Engine program using Process Scheduler.
    • Ensure the program runs at the desired intervals to generate and distribute the reports.

    Query Implementation

    1. Define query profiles
    2. build query access group trees
    3. work with query trees
    4. Define row level security and query security records

    row level security
    With row-level security, users can have access to a table without having access to all rows on that table
    to apply row level security you can add view with SQL condition in record properties of the record which you want to apply row level security


    To implement row-level security through a security view:
    In Application Designer, insert one of the three row-level security fields (OPRID, OPRCLASS, ROWSECCLASS) into the record definition.
    Configure the field as a Key, but not a List Box Item.
    Save the record and build the view.
    Use the record as the search record or query security record.
    Now, when the user searches, the system dynamically adds a WHERE clause — that incorporates the security field — to the search SELECT statement. The value of the security field is based on the current user.

    Query Access Group
    Every record definition that you want users to be able to query must be in a query tree.
